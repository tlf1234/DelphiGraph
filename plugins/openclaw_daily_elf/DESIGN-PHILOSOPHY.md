# 设计理念：边界与职责分离

## 核心原则

### 插件的职责
✅ **发起任务要求**  
✅ **接收分析结果**  
✅ **保存结果到本地**  

### OpenClaw 的职责
✅ **使用它自己的所有工具获取信息**  
✅ **分析和总结信息**  
✅ **生成预测结论**  

## 为什么这样设计？

### 1. 边界清晰
- 插件不应该越界访问 OpenClaw 的内部数据（如 `~/.openclaw/memory/`）
- OpenClaw 有完整的工具生态系统，应该让它自己使用
- 各司其职，避免耦合

### 2. 能力最大化
OpenClaw 能接触到的信息包括：
- 📝 **本地记忆**：历史对话、交互记录、知识库
- 👤 **用户画像**：行为模式、偏好、专业领域
- 🌐 **公网信息**：最新新闻、研究报告、市场数据
- 📊 **历史数据**：时间序列、趋势分析、统计信息
- 🔧 **其他工具**：文件系统、计算工具、API 调用等

让 OpenClaw 自己使用这些工具，能获得最全面的信息。

### 3. 灵活性
- 插件只需要发送任务描述
- OpenClaw 根据任务自主决定使用哪些工具
- 不需要插件预先知道需要什么信息

## 提示词设计

我们的提示词明确指示 OpenClaw 使用所有工具：

- 本地信息获取（记忆检索工具）
- 用户画像分析
- 互联网信息检索（网络搜索工具）
- 历史数据分析
- 综合信息整合

## 信息流

```
用户任务 → 插件发起 → OpenClaw使用所有工具 → 生成结果 → 插件接收 → 保存本地
```

## 对比

### ❌ 错误的做法
直接读取 OpenClaw 的内部文件（边界违规）

### ✅ 正确的做法
通过提示词指示 OpenClaw 使用它的工具

## 总结

**我们不是 OpenClaw 的一部分，我们是 OpenClaw 的用户。**

我们通过 WebSocket 发送任务，OpenClaw 使用它的所有能力完成任务，然后返回结果给我们。
