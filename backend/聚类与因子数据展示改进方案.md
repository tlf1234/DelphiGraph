# 聚类与因子数据展示改进方案

## 问题诊断

### 当前数据流

```
原始预测数据（100条）
    ↓
Phase 1: 信号预处理
    ├─ 输出1: 3-8个主题聚类 ✅ 已生成
    │   ├─ 聚类1: "美联储政策" (25条信号)
    │   ├─ 聚类2: "通胀数据" (18条信号)
    │   └─ 聚类3: "市场情绪" (30条信号)
    │
    └─ 输出2: 处理后的信号列表 ✅ 已生成
    ↓
Phase 2: 因果本体生成
    └─ 输出: 5-8个因果因子 ✅ 已生成
        ├─ 因子1: "货币政策"
        ├─ 因子2: "宏观流动性"
        └─ 因子3: "投资者情绪"
    ↓
Phase 3: 图谱构建
    └─ 输出: 因果图谱（节点+边）✅ 已生成
    ↓
数据库保存
    ├─ graph_data: 因果图谱 ✅ 已保存
    ├─ ontology_data: 因果本体 ✅ 已保存
    └─ preprocess_summary: 预处理摘要 ✅ 已保存
        ├─ clusters: [...]  ⚠️ 已保存但前端未展示
        └─ minority_clusters: [...]  ⚠️ 已保存但前端未展示
    ↓
前端展示
    ├─ 因果图谱可视化 ✅ 已展示（只显示因子节点）
    ├─ 主题聚类视图 ❌ 缺失
    └─ 原始信号列表 ❌ 缺失
```

### 核心问题

1. **数据已保存但未展示**：`preprocess_summary.clusters` 包含完整的聚类信息，但前端没有读取和展示
2. **层级关系不清晰**：用户无法理解"聚类"和"因子"的区别和关联
3. **溯源链断裂**：无法从因子追溯到聚类，再追溯到原始信号

```json
  "sentiment": "negative",
  "anchor_entities": ["美联储", "鲍威尔", "加息"],
```

**特点**：
- 数据层面的分组
- 包含具体的信号集合

- 有情感倾向标注


### 方案 B：在因子节点中关联聚类信息

#### 后端改动

修改 `CausalNode` 模型，增加关联聚类的字段：

```python
@dataclass
class CausalNode:
    # ... 现有字段 ...
    
    # 新增：关联的聚类信息
    source_clusters: List[Dict] = field(default_factory=list)
    # 格式：[{"cluster_id": "...", "theme": "...", "signal_count": 15}]
```

修改 `CausalGraphBuilder._bind_evidence` 方法，记录聚类到因子的映射：

```python
def _bind_evidence(self, graph: CausalGraph, preprocess_result: PreprocessResult) -> None:
    """将预处理聚类中的线索关联到最匹配的因子节点"""
    for cluster in preprocess_result.clusters:
        best_node = self._find_best_matching_node(
            cluster.theme, cluster.anchor_entities, graph.nodes
        )
        if not best_node:
            continue
        
        # 新增：记录聚类到因子的映射
        best_node.source_clusters.append({
            "cluster_id": cluster.cluster_id,
            "theme": cluster.theme,
            "sentiment": cluster.sentiment,
            "signal_count": len(cluster.signals),
            "hard_fact_count": cluster.hard_fact_count,
            "persona_count": cluster.persona_count,
        })
        
        # 原有逻辑：绑定信号
        for sig in cluster.signals:
            best_node.evidence_ids.append(sig.signal_id)
            # ...
```

#### 前端改动

在因子节点详情弹窗中显示来源聚类：

```tsx
<FactorDetailModal factor={selectedFactor}>
  <FactorHeader>

        <SignalCount>{cluster.signal_count}条信号</SignalCount>
        <SentimentIcon sentiment={cluster.sentiment} />
      </ClusterChip>
    ))}
  </SourceClusters>
  
  <EvidenceTraceability>
    <SectionTitle>证据溯源</SectionTitle>
    <SignalList signalIds={factor.evidence_ids} />
  </EvidenceTraceability>
</FactorDetailModal>
```

### 方案 C：5层图谱结构（长期方案）

在图谱可视化中增加"聚类层"节点：

```
当前4层结构：
Agent → Signal → Factor → Target

改进5层结构：
Agent → Signal → Cluster → Factor → Target
                    ↑
                新增层级
```

#### 优点
- 完整展示数据处理流程
- 清晰的层级关系
- 可视化聚类到因子的映射

#### 缺点
- 图谱复杂度增加
- 可能影响可读性
- 需要重构前端图谱渲染逻辑

## 推荐实施路径

### 阶段1：快速修复（1-2天）

1. ✅ **修改后端**：在 `CausalNode` 中增加 `source_clusters` 字段

3. ✅ **修改前端**：在因子详情弹窗中显示来源聚类

5. ✅ **新增信号视图**：创建原始信号列表视图
6. ✅ **增加交互**：点击聚类可跳转到相关因子，点击因子可查看来源聚类

**效果**：用户可以从多个维度查看分析结果

### 阶段3：优化体验（长期）

7. ⏳ **5层图谱**：如果用户反馈需要，考虑实现5层图谱可视化
8. ⏳ **聚类对比**：支持多个聚类的对比分析
9. ⏳ **画像钻取**：支持按人群画像维度筛选聚类和信号

## 用户价值

### 改进前
- ❌ 用户只能看到抽象的因果因子
- ❌ 不知道因子是如何从原始数据中提炼出来的
- ❌ 无法理解"聚类"和"因子"的区别

### 改进后
- ✅ 用户可以看到完整的数据处理流程
- ✅ 可以从因子追溯到聚类，再追溯到原始信号
- ✅ 理解"聚类是数据整理，因子是概念抽象"
- ✅ 可以验证分析结果的可信度

## 总结

**核心问题**：聚类数据已保存但未展示，导致用户无法理解完整的分析流程。

**解决方案**：
1. 短期：在因子详情中显示来源聚类（方案B）
2. 中期：增加独立的聚类视图标签页（方案A）
3. 长期：考虑5层图谱可视化（方案C）

**预期效果**：用户可以从多个维度查看分析结果，理解数据处理的完整流程，提升对分析结果的信任度。
